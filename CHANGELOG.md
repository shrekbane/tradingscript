- `feat(alerts)`: add pip distances to every alert message. The main Buy/Sell signal message now shows "(N pips)" next to TP1/TP2/SL, and the TP1/TP2/TP3/Stop Loss/Breakeven hit messages each show the pip gain (or loss, prefixed "-") from the original entry. Added `pipSize()`, a broker-style convention since Pine has no single builtin for pip size (`syminfo.mintick` is the smallest price increment, not necessarily a real pip — e.g. 5-decimal EURUSD quotes have a mintick 1/10th the size of an actual pip): 0.0001 for standard forex pairs, 0.01 for JPY pairs and for XAUUSD (a common gold convention where 100 pips = $1), and 1.0 for crypto pairs. The crypto figure is a rough, non-standard approximation — pips aren't really a crypto concept — adopted anyway so every pair on the followed-pairs list gets a consistent pip figure in the same alert format, rather than mixing pips and raw price distance depending on instrument.
- `feat(risk)`: add "Use Structure-Based SL/TP" toggle (off by default, untested against real charts yet unlike the S/R confirmation gate) — instead of a fixed ATR distance, SL is placed just beyond the nearest qualifying swing point behind entry, and TP1-3 at the next three swing points ahead of entry, in increasing distance. Added a rolling history of recent swing highs/lows (`swingHighs`/`swingLows`, capped at 30 each) alongside the existing single-most-recent-zone tracking used by the S/R confirmation gate, since that gate only ever needs the latest pivot but this needs several at different distances. New `nearestBeyond()` helper finds the closest swing point on the correct side of entry that's at least "Min Level Distance" (default 0.5x ATR) beyond the previous level (entry itself for SL/TP1, the prior TP for TP2/TP3, so TP1 < TP2 < TP3 can't end up out of order) and no further than "Max Level Distance" (default 8x ATR) from entry, so an existing-but-unrealistically-distant swing point doesn't produce an oversized stop/target. Each of the 4 levels falls back to its existing ATR-based distance independently whenever no qualifying swing point is found for that specific level (too close, too far, or none available) — a `clampBeyond()` helper keeps an ATR-fallback TP from landing out of order relative to whatever level (structural or fallback) it's stacking on top of. Purely opt-in for now; ATR-based sizing remains the default and is not being deprecated — this is an alternative, not a replacement, until validated against real setups the same way the S/R gate was.
- `fix(grading)`: fix HTF Bias and MACD Histogram hard gates silently not holding for signals that spend time in the S/R delayed-confirmation pending state. Previously, `htfTrendOk(direction)`/`macdOk(direction)` were only checked once, at the bar a candidate first appeared (when `pendingDir` gets set) — never re-checked at the later bar the S/R reject actually confirms it, which can be up to `srConfirmBars` bars afterward. Since HTF Bias and MACD Histogram are each just one of five factors feeding the overall score, the score's bull/bear winner (`direction`) can stay unchanged even after one of them individually flips against it, if the other factors (RSI/MACD/ADX) outweigh it — so a candidate queued while HTF was genuinely bullish could still fire later after HTF had already flipped bearish, silently breaking the "unconditional hard gate" guarantee for any signal that spent time waiting. Caught via a real chart case: a BUY A fired with the Data Window showing HTF Bias at -2.000 (fully bearish) and Direction at +1.000 on the same bar, confirmed by hovering the exact signal bar rather than the live/current bar (which was a red herring the first time, since the Info Table only ever reflects the latest bar, not whichever bar is being reviewed in the Data Window). Fixed by re-checking `htfTrendOk(pendingDir) and macdOk(pendingDir)` on every bar while a candidate is pending, dropping it immediately (`pendingDir := 0`) the moment either gate no longer holds, rather than waiting for the window to lapse or letting it fire on stale confluence. Only affects setups where `srGateOn` is on and a candidate takes more than 1 bar to get its reject — same-bar confirmations were never affected.
- `docs(pages)`: simplify checklist item 1 in `README.md` and `index.html` from "check the Data Window factor values, if one or two read near zero" to "check the Trend Strength row in the info table, if it reads 'Weak'" — Trend Strength (ADX/DMI) turns out to be the best single factor to point users at: it's a plain binary read (real trend = full points, no trend = zero, no in-between like RSI's continuous scaling), it isn't one of the hard-gated factors (HTF Bias and MACD are unconditional requirements now, so they're guaranteed to already agree on any fired signal and checking them in the Data Window would never show a red flag), and — unlike the rest of the factor breakdown — it's already surfaced in the always-visible Info Table's "Trend strength" row as "Strong"/"Weak", so this no longer requires hovering the Data Window at all.

- `feat(grading)`: add a Support/Resistance polarity-flip confirmation gate ("Require S/R Flip Reject to Confirm Entry", on by default), plus an optional zone-line plot on the chart (on by default). Tracks the single most recent confirmed pivot high/low (`ta.pivothigh`/`ta.pivotlow`, configurable lookback via "Pivot Lookback"). Once a confirmed close breaks through a tracked zone it "flips" polarity — old resistance becomes a candidate support (RBS), old support becomes a candidate resistance (SBR) — and a "reject" is any later bar that wicks back into the flipped zone (within `atrVal * 0.25` tolerance) but still closes back away from it. Implemented as delayed confirmation, not a same-bar hard requirement like HTF/MACD: a normal signal candidate (score, HTF, MACD, Core Signal, etc. all still apply unchanged) gets locked in as "pending" (direction + grade captured from that bar) rather than firing immediately, and only actually fires — alert, entry line, badge, SL/TP — once a matching reject shows up within "Confirmation Window" bars (default 10). If the window lapses with no reject, the candidate is dropped silently and, since `lastDir` was never advanced, a fresh attempt restarts immediately on the next bar as long as the underlying conditions still hold. Chose delayed confirmation over a same-bar hard requirement because the intended use is "confluence lines up first, then wait for price to confirm it" rather than requiring both to land on the exact same bar. Entry price is deliberately left at the confirmation bar's close (not the zone price itself) — the zone price is, by definition, a level the reject candle already closed away from, so pinning entry there would mean showing a price that's already in the past by the time the alert fires, the same stale-price problem flagged earlier with the GBPUSD alert-vs-chart mismatch. Only tracks one zone at a time in each direction (not a history of every past pivot) as a deliberate first-pass simplification; explored and shelved a same-bar hard-requirement variant and a "require multiple rejects" variant (both discussed but not built) in favor of this simpler version, matching the recommended designs. Started as diagnostics-only (`Diag: S/R Flip Reject` in the Data Window, gate/plot both defaulted off) and was only turned into an actual gate — and flipped on by default — after reviewing several real chart setups where RBS/SBR rejects consistently lined up right after the signals that went on to confirm. An earlier, differently-scoped "Avoid Entries Near Support/Resistance" feature (a same-bar proximity-avoidance filter, not a flip/reject confirmation) was fully reverted in an earlier session after a runtime bug — this is an independent design, not a restoration of that one. A separate engulfing-candle diagnostic was also explored on the way here (added, reviewed against real setups, then explicitly scrapped at the user's request since a single engulf wasn't a reliable enough signal on its own) and left no trace in the current code.

Fixed by deleting the existing SL line/label and creating fresh ones at the breakeven price instead of modifying in place, which isn't subject to the same limit. Added `macdOk(dir) => (dir == 1 and macdHist > 0) or (dir == -1 and macdHist < 0)` right after `macdHist` is computed, folded into `newSignal` alongside `passesFilter`/`corePasses`/`htfTrendOk`/`canOverride`. Prompted by comparing factor diagnostics across 8 real setups (4 B-grade, 4 A+/A-grade): every single B-grade example had MACD Histogram fully opposed (contributing zero to the winning side) while Local Trend, HTF Bias, and ADX/DMI were all maxed in agreement — the bigger picture consistently agreed but the most immediate momentum read hadn't turned yet. Every A+/A example already had MACD fully agreeing naturally, with no gate forcing it, so going unconditional isn't expected to cost much at the top end based on that sample. No new input added — MACD Histogram is scored as a flat binary (2 or 0 points, no partial credit like RSI's distance-scaling), so "requiring agreement" is a plain condition, not a configurable minimum-strength threshold like the earlier (reverted) MACD gate attempt used. Documented in `README.md` (new paragraph following the HTF hard-requirement one) and `index.html` (new callout following the HTF one).
