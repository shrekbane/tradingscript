# Changelog

A commit backlog for GradeRunner — one line per change, newest at the top, grouped by date. Each line is a `type(scope): summary` in the same style as a real commit message, just written after the fact from session history rather than from actual diffs.

## 2026-07-24

- `feat(grading)`: add a Support/Resistance polarity-flip confirmation gate ("Require S/R Flip Reject to Confirm Entry", on by default), plus an optional zone-line plot on the chart (on by default). Tracks the single most recent confirmed pivot high/low (`ta.pivothigh`/`ta.pivotlow`, configurable lookback via "Pivot Lookback"). Once a confirmed close breaks through a tracked zone it "flips" polarity — old resistance becomes a candidate support (RBS), old support becomes a candidate resistance (SBR) — and a "reject" is any later bar that wicks back into the flipped zone (within `atrVal * 0.25` tolerance) but still closes back away from it. Implemented as delayed confirmation, not a same-bar hard requirement like HTF/MACD: a normal signal candidate (score, HTF, MACD, Core Signal, etc. all still apply unchanged) gets locked in as "pending" (direction + grade captured from that bar) rather than firing immediately, and only actually fires — alert, entry line, badge, SL/TP — once a matching reject shows up within "Confirmation Window" bars (default 10). If the window lapses with no reject, the candidate is dropped silently and, since `lastDir` was never advanced, a fresh attempt restarts immediately on the next bar as long as the underlying conditions still hold. Chose delayed confirmation over a same-bar hard requirement because the intended use is "confluence lines up first, then wait for price to confirm it" rather than requiring both to land on the exact same bar. Entry price is deliberately left at the confirmation bar's close (not the zone price itself) — the zone price is, by definition, a level the reject candle already closed away from, so pinning entry there would mean showing a price that's already in the past by the time the alert fires, the same stale-price problem flagged earlier with the GBPUSD alert-vs-chart mismatch. Only tracks one zone at a time in each direction (not a history of every past pivot) as a deliberate first-pass simplification; explored and shelved a same-bar hard-requirement variant and a "require multiple rejects" variant (both discussed but not built) in favor of this simpler version, matching the recommended designs. Started as diagnostics-only (`Diag: S/R Flip Reject` in the Data Window, gate/plot both defaulted off) and was only turned into an actual gate — and flipped on by default — after reviewing several real chart setups where RBS/SBR rejects consistently lined up right after the signals that went on to confirm. An earlier, differently-scoped "Avoid Entries Near Support/Resistance" feature (a same-bar proximity-avoidance filter, not a flip/reject confirmation) was fully reverted in an earlier session after a runtime bug — this is an independent design, not a restoration of that one. A separate engulfing-candle diagnostic was also explored on the way here (added, reviewed against real setups, then explicitly scrapped at the user's request since a single engulf wasn't a reliable enough signal on its own) and left no trace in the current code.
- `fix(risk)`: fix a `modify_study_limit_exceeding` runtime error in the breakeven-after-TP1 feature — the `hitTP1` block was moving the drawn SL line/label to breakeven in place via `line.set_y1`/`line.set_y2`/`label.set_y`/`label.set_text`, but Pine caps the total number of drawing *modifications* across a script's full history recalculation much lower than the number of objects it allows creating/deleting, and enough historical TP1 hits eventually exceeded that cap. Fixed by deleting the existing SL line/label and creating fresh ones at the breakeven price instead of modifying in place, which isn't subject to the same limit. Added `var int entryBarIndex` (set alongside `entryPrice` in the `newSignal` block) so the recreated line/label use the original entry bar's x-position (`entryBarIndex`/`entryBarIndex + extendBars`) rather than the TP1-hit bar's, keeping them visually aligned with the entry/TP lines exactly as before. No change to the feature's behavior or wording (still relabels to `"BE " + price`, still moves `slPrice` so `hitSL` checks the new level) — this is a rendering-mechanism fix only.
- `feat(grading)`: add MACD Histogram as a hard gate, unconditional across every grade, same posture as the HTF Bias gate. Added `macdOk(dir) => (dir == 1 and macdHist > 0) or (dir == -1 and macdHist < 0)` right after `macdHist` is computed, folded into `newSignal` alongside `passesFilter`/`corePasses`/`htfTrendOk`/`canOverride`. Prompted by comparing factor diagnostics across 8 real setups (4 B-grade, 4 A+/A-grade): every single B-grade example had MACD Histogram fully opposed (contributing zero to the winning side) while Local Trend, HTF Bias, and ADX/DMI were all maxed in agreement — the bigger picture consistently agreed but the most immediate momentum read hadn't turned yet. Every A+/A example already had MACD fully agreeing naturally, with no gate forcing it, so going unconditional isn't expected to cost much at the top end based on that sample. No new input added — MACD Histogram is scored as a flat binary (2 or 0 points, no partial credit like RSI's distance-scaling), so "requiring agreement" is a plain condition, not a configurable minimum-strength threshold like the earlier (reverted) MACD gate attempt used. Documented in `README.md` (new paragraph following the HTF hard-requirement one) and `index.html` (new callout following the HTF one).
- `feat(diagnostics)`: re-add per-factor score diagnostics to TradingView's Data Window, prompted by wanting to actually verify (rather than just theorize) why B-grade signals were showing a low TP1-before-SL rate in the Grade Win-Rate Stats panel — hypothesis was that B-grade signals disproportionately have Local Trend and/or ADX/DMI Strength contributing nothing, relying mainly on HTF Bias (now guaranteed by the hard gate) plus RSI/MACD to barely clear the B threshold. Refactored Local Trend, HTF Bias, and MACD Histogram's bull/bear points (previously inlined directly into `bullPts`/`bearPts`) into named `trendBullPts`/`trendBearPts`/`htfBullPts`/`htfBearPts`/`macdBullPts`/`macdBearPts` variables, matching the style RSI already used — purely cosmetic, same values computed either way. Hoisted `adxBullPts`/`adxBearPts` to be declared (defaulting to `0.0`) before the `if adxStrong` block instead of only inside it, switching their assignment to `:=`, so they're readable outside that scope too — behavior is identical, since they still land on `0.0` whenever `adxStrong` is false. Added 7 new plots, all `display=display.data_window` (invisible on the chart/price-scale/legend, visible only when hovering a bar in the Data Window): the net point contribution (bull minus bear, −2..+2 range) for each of the 5 confluence factors, plus the overall `score` and `direction`. This is a same-shape reimplementation adapted to the current 5-factor version (not a restoration from git history — the original had 6 factors, including the since-reverted VWAP) — no other reverted feature was brought back alongside it. Documented in `README.md` (new "Explaining a past signal's grade" paragraph) and `index.html` (new feature grid card).
- `feat(grading)`: re-add HTF Bias as a hard gate, this time unconditional across every grade (no A/A+ carve-out like the earlier attempt). Added `htfTrendOk(dir) => (dir == 1 and htfBull) or (dir == -1 and not htfBull)` right after `htfBull` is computed, and folded it into `newSignal` alongside `passesFilter`/`corePasses`/`canOverride`. Reframed around a specific, deliberately-stated goal this time — not "improve win rate" (the framing the earlier attempt was judged against and reverted over), but stopping a specific known trading habit: catching a bounce against the prevailing higher-timeframe trend, prompted by a live XAUUSD example where a Grade A BUY fired on a sharp bounce candle while HTF Bias read Bearish, then pulled back immediately. A carve-out for A/A+ would have let that exact example straight through, since a strong bounce candle can score high enough to hit A on its own — so this time the gate applies everywhere, accepting that it will also block some genuinely good counter-trend reversals as a deliberate trade-off for the discipline, rather than trying to optimize signal count or raw win rate. Documented in `README.md` (new paragraph under "How it works" explaining the hard-requirement framing) and `index.html` (new "always on, not just scored" callout under the grade table).

- `feat(risk)`: move the stop to breakeven automatically once TP1 hits, and rename the resulting hit to "BE hit"/"Breakeven hit" instead of "SL hit"/"Stop Loss hit." In the `hitTP1` block, `slPrice` is reassigned to `entryPrice`, and — if `showLabels` is on — the already-drawn SL line/label are moved in place via `line.set_y1`/`line.set_y2`/`label.set_y`/`label.set_text` (relabeled `"BE " + price`) rather than left stale, so the chart never shows a stop that no longer matches what the script is actually tracking. Since `hitSL`'s price check reads the live `slPrice` each bar, it automatically evaluates against the new breakeven level with no separate condition needed. In the `hitSL` block, wording (both the hit tag and the `alert()` message) branches on `tp1Hit`: `"SL hit"`/`"Stop Loss hit"` if TP1 never fired, `"BE hit"`/`"Breakeven hit"` if it did — so a flat exit after already banking TP1 isn't misread as a loss. The TP1 hit `alert()` message also now confirms the move (e.g. "TP1 hit @ 4029.545 — stop moved to breakeven (4020.975)"). Chose to actually move the tracked stop rather than fire an advisory-only reminder, since a reminder only protects the trade if the user is present to act on it in time, and a manually-moved stop the script doesn't know about would create the same kind of chart/alert drift just fixed with the multi-line message. No change to Grade Win-Rate Stats logic — a loss is still only tallied `if not tp1Hit`, which already matches this behavior exactly (a post-TP1 breakeven exit was never counted as a loss to begin with). The named "Stop Loss Hit" alert condition (picked individually in Create Alert, not "Any alert() function call") keeps its generic TradingView-placeholder wording regardless, same as the other named conditions. Documented in `README.md` (new "Breakeven after TP1" section, and an update to the alerts explanation) and `index.html` (new feature card, updated TP1 Hit preview, and a note in the Alerts section).
- `feat(risk)`: change default TP multipliers from 1x/2x/3x ATR to 2x/3x/4x ATR (SL unchanged at 1x), so TP1 lands at 2:1 risk-reward instead of 1:1 — TP2/TP3 shifted up by the same +1x ATR to preserve the even 1x-ATR spacing between targets rather than letting TP1 collide with the old TP2 level. Purely default-value changes to existing `tp1Mult`/`tp2Mult`/`tp3Mult` inputs — still freely adjustable per-instrument in settings. Updated `README.md` and `index.html`'s risk-reward figures and example alert prices to match the new ratios.
- `feat(alerts)`: re-add the multi-line Buy/Sell `alert()` message (Entry/TP1/TP2/SL each on their own line, TP3 omitted) on top of the commit-6db787b baseline, prompted by a live case where a signal's alerted entry price (captured from the real-time close at the moment of firing) differed slightly from the entry price the chart later redrew after a reload (recalculated against the data vendor's settled historical close for that bar) — the multi-line message means the exact levels that actually fired are always readable straight from the notification, independent of the chart's redraw. Documented in `README.md` (updated alert preview examples, and a new note explaining why the alert message is the authoritative record vs. the chart's drawn lines) and `index.html` (updated Buy/Sell notification preview cards to the multi-line format, plus a matching TP3-omitted note) — caught before it could repeat the earlier miss where `index.html` was forgotten when this same format was first added.
